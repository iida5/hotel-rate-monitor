# 📡 ホテル競合レート監視ダッシュボード

競合ホテル群 × 日付グリッド (リードタイム別のチェックイン日) の「部屋タイプ別価格」と「残室数」を
定点観測し、変動した瞬間にアラートを出すレートショッパーです。

データ収集は3方式から選べます (既定は**ブラウザ収集**):

- **ブラウザ収集** (`browser`) — Playwright で楽天トラベルの各ホテルページを開いて読み取る。**APIキー不要**で、APIが廃止されても動き続ける。残室数も「残りX室」表示から実値で取得
- **楽天トラベルAPI** (`rakuten`) — 公式APIで取得 (要アプリID/アクセスキー)
- **シミュレーション** (`sim`) — 架空データのデモ

## 起動方法

初回のみブラウザ収集用の Chromium を取得します:

```powershell
npm install
npx playwright install chromium
```

以降は起動するだけです:

```powershell
node server.js
```

ブラウザで **http://localhost:3000** を開いてください。

## 画面からの設定変更

以下の5項目は、画面右上の **「設定」** ボタンから変更できます。保存すると監視が自動で再起動し、
内容は `data/rateshop-config.json` に保存されます (環境変数より優先。再起動後も保持されます)。

- **監視施設** — 楽天トラベルの施設番号 (1行に1件 / カンマ区切り)
- **自社施設** — 監視施設のうち1つ。設定するとグリッドの競合セルに**自社との差額**、自社行に**価格順位**が出る
  (自社変動を通知から除外する判定にも使用)
- **監視するチェックイン日** — 今日から何日後か (`0`=今夜。例 `0,1,2,3,7,14,30`、最大14件)
- **空室検索する宿泊人数** — カンマ区切りで**最大3条件を並行監視** (例 `1,2` でシングル1名とツイン2名の両需要を監視)
- **ブラウザの並列ページ数** (1〜10)

設定変更後の最初の1巡はベースラインの取り直しになるため、その間アラートは出ません。

## 環境変数 (初回起動時の既定値)

`.env` ファイルに書けます (`.env.example` をコピーして編集。シェルの環境変数が優先)。
上記5項目については、画面から保存した内容があればそちらが優先されます:

```powershell
copy .env.example .env
notepad .env   # APIキー・監視対象を記入して保存
node server.js
```

| 環境変数 | 既定値 | 説明 |
|---|---|---|
| `PORT` | `3000` | リッスンポート |
| `HOST` | `127.0.0.1` | バインド先アドレス |
| `RAKUTEN_APP_ID` | (なし) | 楽天トラベルAPIのアプリID |
| `RAKUTEN_ACCESS_KEY` | (なし) | 楽天APIのアクセスキー。**2026年2月のAPI刷新以降は必須** |
| `RAKUTEN_ORIGIN` | (なし) | アプリ登録した「許可されたWebサイト」のドメイン。403エラーが出る場合に設定 (例 `https://github.com`) |
| `RATESHOP_HOTELS` | (なし) | 監視する楽天施設番号のカンマ区切り。未設定ならシミュレーションモード |
| `RATESHOP_OWN` | (なし) | `RATESHOP_HOTELS` のうち自社の施設番号 |
| `RATESHOP_DATES` | `1,3,7,14,30` | 監視するチェックイン日 (今日から何日後か) |
| `RATESHOP_ADULTS` | `2` | 空室検索する宿泊人数。カンマ区切りで最大3条件 (例 `1,2`) |
| `RATESHOP_STOCK_PROBE` | `5` | 残室数プローブの最大段数 (1〜10)。1で無効 |
| `RATESHOP_TICK_MS` | `5000` | シミュレーションモードの更新間隔 (ミリ秒) |
| `RATESHOP_WEBHOOK` | (なし) | アラート送信先の Slack 互換 Webhook URL |
| `RATESHOP_WEBHOOK_TYPES` | `price_up,price_down,soldout,restock` | Webhook で送る種別 (在庫増減は既定で送らない) |
| `RATESHOP_WEBHOOK_PCT` | `0` | Webhook で送る価格変動の最小% (例 `5` で±5%以上のみ) |

## 監視単位と検知できる変動

実データモードの監視単位は **「宿泊人数プロファイル × 部屋タイプ (roomClass) 別の最安値」** です。
プランごと検索 (安い順・最大60プラン) を部屋タイプ別に集約するため、
「ツイン同士で自社と競合はいくら差があるか」という同条件比較ができ、
SALEプランの終了などによる入れ替わりは「その部屋タイプの最安値が変わった」
という1件の価格アラートに集約されます (そのとき最安のプラン名も併記)。
人数を複数指定 (例 `1,2`) すると、シングル1名のビジネス需要と2名利用の需要を
別々の条件として監視・アラートし、グリッド上部のタブで表示を切り替えられます。

**残室数の表記**: 楽天サイトの「残りX室」バッジから取れた**実値**はそのまま「残3室」、
在庫プローブによる**推定値**は「残約3室」と「約」付きで表示します (アラートも同様)。
推定値は「予約画面で一度に予約できた室数」なので、施設の販売制限により実在庫より少なく出ることがあります。

| アラート種別 | 検知内容 |
|---|---|
| 値上げ / 値下げ | 部屋タイプ別最安値の変動 (変動率%付き。最安プランの入れ替わりを含む) |
| 売止/満室 | 販売中だった部屋タイプが予約不可になった |
| 販売再開 | 売止だった部屋タイプの販売再開 |
| 在庫増 / 在庫減 | 部屋タイプ別の残室数 (予約可能数の推定値) の増減 |

## 主な機能

- **価格比較グリッド (自社ポジション表示)** — ホテル×日付で最低価格と残室を一覧表示。自社施設を設定すると、
  競合セルに自社との差額 (自社より安い競合は橙色で強調)、自社行に価格順位 (n位/m軒、最安値同士の比較) を表示。
  金土の列は色分け。ホテル名クリックで部屋タイプ別に展開。変動したセルは一瞬光る
- **推移チャート** — 価格セルをクリックすると、そのホテル×日付の最安値と残室の推移をチャート表示
  (部屋タイプの内訳行のセルならその部屋タイプ単独の推移)。チャートから楽天の該当ページも開ける。
  観測履歴は `data/rateshop-observations.*.ndjson` に自動蓄積され、Excel等での時系列分析にも使える
- **鮮度表示と収集停止の警告** — 各セルは最終観測から30分を超えると淡色表示になり (ツールチップに観測時刻)、
  グリッド全体の観測が30分以上止まると赤い警告を表示。古い価格を現在値と誤認して値付けする事故を防ぐ
- **アラートに自社最安を併記** — 競合の値上げ/値下げアラートには同日付・同人数の自社最安値を添える
  (通知だけで「追随するか流すか」を判断できる)。実データモードでは各アラートから楽天の該当ページに1クリックで飛べる
- **前回確認時からの要約** — 画面を開くと「前回確認から: 値下げ2件・売止1件」のような要約をアラート欄の先頭に表示
  (在庫増減のような細かい通知に大事な変動が埋もれない)
- **観測インジケータ** — 「いまどのホテルのどの日付を観測しているか」とサイクル進捗をレーダー風に表示。観測したセルも青く光る
- **アラートフィード** — 種別チップ・価格変動しきい値 (±3/5/10%)・自社表示の有無でフィルタ。設定はブラウザに保存。
  **既定は「価格±5%以上・売止・販売再開」のみ表示** (在庫増減はチップでONにできる)
- **デスクトップ通知** — 🔔ボタンでONにすると、フィルタ条件に合う新着アラートをOS通知で表示
- **Webhook通知** — `RATESHOP_WEBHOOK` にSlack互換のIncoming Webhook URLを設定すると、競合の変動をPOST
  (自社の変動は送らない。既定で在庫増減は送らず、`RATESHOP_WEBHOOK_TYPES` / `RATESHOP_WEBHOOK_PCT` で調整可)
- **履歴の永続化** — 全アラートを `data/rateshop-alerts.*.ndjson` に追記 (時系列分析にそのまま使える)
- **KPIカード** — 直近1時間の価格改定数 / 売止セル数 / 直近泊の競合最安 (自社との差額付き) / 24時間のアラート数

## モード

1. **シミュレーション (既定)** — 新宿エリアの架空競合8軒×4プラン×5日付を確率的に変動させるデモ。起動するだけで動作確認できる
2. **楽天トラベルAPI (実データ)** — `RAKUTEN_APP_ID`・`RAKUTEN_ACCESS_KEY` と `RATESHOP_HOTELS` を設定すると、
   `VacantHotelSearch` でプラン別実勢価格と残室数を巡回観測する

```ini
# .env の例: 競合3軒 (うち 143637 が自社) を監視
RAKUTEN_APP_ID=あなたのアプリID
RAKUTEN_ACCESS_KEY=あなたのアクセスキー
RATESHOP_HOTELS=143637,168223,182311
RATESHOP_OWN=143637
```

> **2026年2月のAPI刷新について**: 楽天ウェブサービスはエンドポイントが `openapi.rakuten.co.jp` に変わり、
> `applicationId` に加えて `accessKey` が必須になりました (旧ドメインは2026年5月14日に完全停止)。
> キーは [アプリリスト](https://webservice.rakuten.co.jp/app/list) で確認できます。
> 403エラーが出るときは、アプリ登録した「許可されたWebサイト」のドメインを `RAKUTEN_ORIGIN` に設定してください。

### ブラウザ収集モードの仕様 (`RATESHOP_SOURCE=browser` / 既定)

- 1ホテル×1日付あたり「1ページ読み込み」で全プラン×全部屋タイプを取得 (画像・フォントは読み込まず高速化)
- 複数ページを並列に開く (`RATESHOP_CONCURRENCY`、既定3)。1巡の目安は「ホテル数 × 日付数 × 約6秒 × プローブ段数 ÷ 並列数」
- **並列数とブロックの注意** — 並列数を上げるほど速くなりますが、同一IPからのアクセスが集中するとボット検知 (CAPTCHA・一時ブロック) を受けやすくなります。
  既定の3前後を推奨し、最大10まで設定可能です
- **残室数** — 2段構えで取得します。(1) 楽天サイトが表示する「残りX室」バッジがあればその**実値**。
  (2) バッジが無い部屋タイプは、予約部屋数 (`f_heya_su`) を2室→3室…と増やしたページを読み
  「一度に何室予約できるか」を推定します (**在庫プローブ**。既定で最大5段、`RATESHOP_STOCK_PROBE` で1〜10に変更可。
  上限まで予約できた部屋タイプは「残5室+」表示)。プローブの分だけ1巡の時間が伸びます
  (5軒×5日付・並列3・5段で1巡 約3〜4分。`RATESHOP_STOCK_PROBE=1` でプローブ無効=バッジ実値のみ・約1分)
- **満室日の検知** — 満室日は部屋行だけ表示され価格が出ないため、「部屋行はあるが価格が1件も無い」状態を全タイプ満室と判定し、
  グリッドに「満室/売止」を表示します。販売が再開されると販売再開アラートが出ます
- 部屋タイプは `data-room-type-img-wrap` 属性のコード (`srw`/`ctw`…) でグルーピングするため、表示名の揺れに影響されず安定します
- 価格は標準の合計価格 (税込) を採用します (クーポン適用後価格は条件付きのため比較には使いません)
- 施設番号は楽天トラベルのホテルページURL (`https://travel.rakuten.co.jp/HOTEL/施設番号/...`) から確認できます
- HTML構造が変わると取得できなくなることがあります (その場合は `lib/browser-source.js` の `parsePageInBrowser` 内セレクタを調整)

### 楽天トラベルAPIモードの仕様 (`RATESHOP_SOURCE=rakuten`)

- 1ホテル×1日付あたり最大「プローブ段数+1」リクエスト。レート制限 (1リクエスト/秒) を守って巡回するため、
  1巡の所要時間は最大「ホテル数 × 日付数 × (プローブ段数+1) × 約1.1秒」(例: 5軒×5日付×6 ≒ 2.8分)
- **残室数の推定 (在庫プローブ)** — 公式APIは残室数を直接返さないため、`roomNum` (部屋数) を1室→2室→…と増やしながら
  照会し、各部屋タイプが「一度に何室まで予約可能か」を残室数の推定値として表示します (既定で最大5段、`RATESHOP_STOCK_PROBE` で1〜10に変更可)。
  上限まで予約できた部屋タイプは「残5室+」のように表示します。1予約あたりの販売上限がある場合は実在庫より少なく出ることがあります

### 共通

- 初回の1巡はベースライン登録のためアラートは出ません。2巡目以降から前回値との差分を検知します
- 観測状態は `data/rateshop-state.<ソース>.json` に保存され、再起動時は前回値との差分からアラートを再開します (誤報防止)

## HTTP API

| エンドポイント | 内容 |
|---|---|
| `GET /api/rateshop/snapshot` | 現在の比較グリッド + 直近アラート (JSON) |
| `GET /api/rateshop/alerts?limit=100` | アラート履歴 (最大400件) |
| `GET /api/rateshop/history?hotel=&date=&adults=` | ホテル×日付の観測履歴 (推移チャートの元データ) |
| `GET /api/rateshop/stream` | SSE リアルタイムストリーム |
